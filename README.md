## 受講者への配布・セットアップ案内

メンターが口頭または別紙で案内する想定の手順です。  
**推奨の起動方法は `setup.bat` です。** 受講者には本 README ではなく、`setup.bat` ごと配布してください。

### 必要環境

- **Windows**（`setup.bat` 利用時）
- Node.js **18 以上**
- npm（Node.js に同梱）

### 起動方法（推奨: setup.bat）

1. 配布された `retro-memory-game` フォルダを任意の場所に置く
2. **`setup.bat` をダブルクリック** する

`setup.bat` は次の処理を自動で行います。

| 順番 | 処理 | 内容 |
|------|------|------|
| 1 | Node.js 確認 | 未インストールの場合はエラーメッセージを表示して終了（`pause`） |
| 2 | 作業フォルダ特定 | `setup.bat` の置き場所から `retro-memory-game` を自動判定 |
| 3 | 依存パッケージ | `node_modules` が **既にある場合は `npm install` をスキップ** |
| 4 | 開発サーバー起動 | 別ウィンドウで `npm run dev` を実行 |
| 5 | ブラウザ起動 | Vite が起動完了後、`http://localhost:5173` を自動オープン |
| 6 | 完了メッセージ | setup.bat のウィンドウに完了表示（`pause`） |

**重要:** `npm run dev` のウィンドウに Vite のログが表示されたまま止まって見えるのは **正常** です（サーバーが起動中の状態）。  
setup.bat のウィンドウには「セットアップ完了」と表示され、**そのウィンドウは閉じて構いません**。

ゲームを終了するときは、タスクバーの **「CRYSTAL MEMORY - Dev Server」** ウィンドウで **Ctrl+C** を押してください。

#### setup.bat の配置場所

次の **どちらか** に置くと動作します。

| 配置 | 例 |
|------|-----|
| プロジェクト直下（配布時の標準） | `retro-memory-game\setup.bat` |
| プロジェクトの親フォルダ | `Projects\setup.bat`（`retro-memory-game` と同階層） |

#### 再インストールが必要な場合

依存パッケージを入れ直したいときは、`retro-memory-game` 内の **`node_modules` フォルダを削除** してから `setup.bat` を再実行してください。

### 起動方法（代替: コマンドライン）

`setup.bat` が使えない環境向けの手順です。

```bash
cd retro-memory-game
npm install
npm run dev
```

2 回目以降、`node_modules` が残っていれば `npm install` は省略可能です。

## SE の基礎として学べること

本教材に意図的に含めている学習ポイントです。メンターは設問と照合して確認してください。

### 1. モジュール分割と責務分離

| ファイル | 責務 |
|----------|------|
| `data/cards.js` | データ定義（キャラ・難易度） |
| `game/engine.js` | ルール・状態遷移（UI 非依存） |
| `ui/render.js` | 見た目の生成 |
| `ui/events.js` | ユーザー操作と各層の接続 |
| `storage/scores.js` | 永続化 |

**学び:** UI とロジックを分けると、テスト・変更がしやすい。

### 2. 配列操作とデータ構造

- デッキ生成（ペアの複製 → シャッフル）
- `Set` によるマッチ済み管理
- スプレッド構文・`map` / `flatMap` / `find`

**学び:** ゲーム状態を「配列 + 補助データ」で表現する考え方。

### 3. 状態管理

- `faceUp` / `flipped` / `matched` / `locked` / `finished` の関係
- 2 枚めくり後の `locked` による入力制御
- `cloneState` による状態コピー

**学び:** 「今何が起きているか」を変数の組み合わせで表す設計。

### 4. イベント処理

- クリックイベントの委譲（`closest('[data-uid]')`）
- 非同期のめくり戻し（`setTimeout`）
- タイマー（`setInterval` / `clearInterval`）

**学び:** ユーザー操作 → 状態更新 → 再描画 の流れ。

### 5. DOM 操作

- テンプレート文字列による HTML 生成
- `innerHTML` による画面切り替え
- CSS クラス（`is-flipped`, `is-matched`）と状態の対応

**学び:** 状態が UI にどう反映されるかを追う。

### 6. 永続化と防御的プログラミング

- `localStorage` の読み書き
- `try/catch` による例外吸収（プライベートモード等）
- 入力値のサニタイズ（`escapeHtml`）

**学び:** 外部ストレージは失敗する前提で書く。

### 7. 定数・設定値の切り出し

- `DIFFICULTY`, `FLIP_DELAY_MS`, `STORAGE_KEY`
- マジックナンバーを避ける意図

## 意図的な「議論ポイント」（模範解答メモ）

ソースにコメントはありませんが、メンター向けに設計上の論点を 3 点入れています。

### 論点 A: 再描画の粒度（`ui/events.js`）

カードをめくるたびに `renderGameScreen` で **画面全体を再生成** しています。

- **現状の意図:** コード量を抑え、状態 → UI の対応を追いやすくする
- **議論:** 部分更新にすると何が楽/大変か。パフォーマンスとのトレードオフ

### 論点 B: タイマーの二重管理（`game/timer.js` + `ui/events.js`）

ゲーム時間用と HUD 更新用で **別々の interval** が動いています。

- **議論:** 1 本にまとめるべきか。`beforeunload` で止めているが他に必要な箇所は？

### 論点 C: シャッフルの実装（`game/engine.js`）

Fisher-Yates 風のシャッフルで、**コピー配列** を返しています。

- **確認課題:** 元配列を直接 `sort(() => Math.random() - 0.5)` した場合との違い

## ワークブック設問

受講者用に別紙化して配布してください。模範解答の要点を `[メモ]` で記載しています。

### 基礎（1〜2 日目）

1. `main.js` から処理が始まり、タイトル画面が表示されるまでの流れを、ファイル名と関数名で説明せよ。  
   `[メモ] main.js → createScreenController → showStart → renderStartScreen`

2. `DIFFICULTY` の `easy` と `normal` では、カード枚数はそれぞれ何枚になるか。計算式も示せ。  
   `[メモ] pairCount × 2。easy=12枚、normal=16枚`

3. `buildDeck` はどのような手順でデッキを作っているか。  
   `[メモ] roster 取得 → 各キャラ2枚生成 → shuffle`

4. `shuffle` は引数の配列そのものを変更しているか。コード上の根拠を示せ。  
   `[メモ] `[...items]` でコピーしてから並べ替え。元配列は変更しない`

5. `canFlip` が `false` を返す条件をすべて列挙せよ。  
   `[メモ] locked / finished / カードなし / 既に表 / 既にマッチ済 / flippedが2枚`

### 応用（3〜4 日目）

6. カードを 1 枚クリックしたとき、どの関数がどの順番で呼ばれるか。  
   `[メモ] handleCardClick → canFlip → flipCard → paintBoard → renderGameScreen`

7. 2 枚めくって不一致だった場合、カードが裏向きに戻るまでに何が起きるか。  
   `[メモ] locked=true → 700ms wait → resolveFlip → faceUp=false → 再描画`

8. 2 枚めくった直後に 3 枚目をクリックできない理由を、変数名を挙げて説明せよ。  
   `[メモ] flipCard で locked=true。canFlip が false`

9. `escapeHtml` は何のために使われているか。使わないとどんな問題が起きうるか。  
   `[メモ] XSS 対策。プレイヤー名に HTML/スクリプトを入れられた場合`

10. `saveRecord` が `false` を返すのはどんなときか。  
    `[メモ] localStorage.setItem 失敗（容量超過・プライベートモード等）`

11. ゲーム中、タイマー表示は何 ms 間隔で更新されているか。根拠のコード位置も示せ。  
    `[メモ] 250ms。events.js の setInterval`

### 発展（5 日目）

12. `engine.js` の `cloneState` はなぜ必要か。なくするとどんな不具合が起きうるか。  
    `[メモ] 状態の意図しない共有・巻き戻し。議論の余地あり`

13. 部分更新（変更カードだけ DOM 更新）にリファクタすると、どのファイルが大きく変わるか。  
    `[メモ] render.js, events.js。engine.js はほぼ不変`

14. ユニットテストを 1 本だけ書くとしたら、どの関数から始めるのがよいか。理由も述べよ。  
    `[メモ] shuffle / resolveFlip / canFlip など純粋関数から`

15. 難易度「HARD（6×4）」を追加するとき、変更が必要なファイルをすべて挙げよ。  
    `[メモ] cards.js（DIFFICULTY, ROSTER 数）、styles.css（レスポンシブ確認）`

| 症状 | ヒントの渡し方 |

## トラブルシューティング

| 現象 | 対処 |
|------|------|
| `setup.bat` 実行時「Node.js がインストールされていません」 | Node.js 18 以上をインストールし、PC を再起動してから再実行 |
| `npm install` 失敗 | Node.js バージョン確認（18+）、ネットワーク接続を確認。`node_modules` を削除して再実行 |
| 2 回目以降も毎回 `npm install` される | `retro-memory-game\node_modules` が存在するか確認（別フォルダで実行していないか） |
| 依存パッケージを入れ直したい | `node_modules` フォルダを削除してから `setup.bat` を再実行 |
| ポート 5173 が使用中 | 他の Vite / Node プロセスを終了するか、`npx vite --port 5174` で手動起動 |
| Ctrl+C 後に「異常終了」と表示される | 手動停止の場合は問題ないことが多い。エラー内容がなければ無視してよい |
| スプライトが表示されない | `public/sprites/` が配布物に含まれているか確認 |
| 記録が保存されない | ブラウザの Cookie/ストレージ設定、プライベートモードを確認 |
